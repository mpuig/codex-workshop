# Appendix B: Guardrails

> **Type:** Reference | **Prerequisites:** None

This appendix focuses on guardrails that are stable and easy to verify in current Codex CLI workflows.

## 1) Start with Safe Execution Defaults

Use conservative defaults for day-to-day work:

```bash
codex -a on-request -s workspace-write
```

For higher-risk environments, prefer read-only until you are ready to apply changes.

```bash
codex -a on-request -s read-only
```

## 2) Keep Durable Instructions in AGENTS.md

Put non-negotiable rules in `AGENTS.md`, for example:

- File/format conventions
- Regulatory terminology requirements
- Forbidden operations
- Review checklist before final output

This is more reliable than repeating the same instructions in chat every session.

## 3) Use Git as a Safety Boundary

A practical review loop:

```bash
git status
git diff
```

Only proceed once changes are understandable and aligned with requirements.

## 4) Enforce Quality with Standard Tooling

Run your project checks explicitly:

```bash
npm test
npm run lint
```

Or the equivalent commands in your stack (`pytest`, `go test`, `cargo test`, etc.).

## 5) Protect Critical Paths by Policy

If certain files should not be touched (for example production pricing models), enforce this in your team process:

- Code owners / mandatory reviews
- Branch protection rules
- CI checks that fail on forbidden paths

## Key Takeaways

- Prefer verified controls (approval policy, sandbox mode, Git, CI) over undocumented internals.
- Keep policy and standards in `AGENTS.md` plus repository governance.
- Make review explicit: inspect diffs and run checks before accepting outputs.
